# PRD Feasibility Review

## Executive Verdict
READY WITH CHANGES

The product is feasible from the available Cardmarket export data, but the PRD is not implementation-ready as-is. The largest gap is not product intent; it is import semantics: XLS Shipment exports are grouped row exports with 8,432 continuation rows where `OrderID` and party fields are blank and must be forward-filled from the previous header row. The PRD and `Datenmodell.md` broadly identify the right entities, but they do not explicitly specify this row-grouping rule, and the MVP scope is larger than necessary for a reliable first build.

## Datenmodell Alignment

The PRD aligns well with `Datenmodell.md` on the broad entity structure: `import_file`, raw export rows, `shipment`, `shipment_event`, `shipment_line`, `product`, `expansion`, `category`, `party`, and address handling. It also correctly adopts the core insight that purchase-date and payment-date files are different views of the same business objects, not independent facts.

The main divergence is operational rather than conceptual. `Datenmodell.md` says `OrderID` and `Shipment nr.` are the same business ID, but neither the data model nor PRD explicitly states that Shipment exports are grouped multirow records where only the first row of an order has `OrderID`, party, address, date, and total fields. Direct XLS evidence shows 8,432 of 10,596 shipment rows have blank `OrderID` and must inherit the previous nonblank order context. Without this rule, the normalized model would drop most shipment-line evidence or create orphan raw rows.

The second divergence is Product naming. `Datenmodell.md` treats `localized_product_name` as a product attribute. Direct XLS evidence shows 63 Product IDs with multiple localized names. That does not invalidate Product ID as key, but the model should store observed localized labels historically or choose a canonical display name with deterministic precedence.

The third divergence is the PRD's MVP breadth. The data model supports a dashboard eventually, but implementation readiness depends first on a robust import pipeline and validation reports. Interactive diagrams, saved reports, saved filters, PDF export, and chart click-to-filter should not be in the first MVP acceptance gate.

## Blindspot Pass

Known knowns:

- Source files are flat, one-sheet exports with stable headers by export type.
- File naming reliably encodes direction, level, date basis, and period.
- Product IDs, article names, shipment numbers, order IDs, amounts, values, comments, costs, usernames, addresses, and VAT fields exist in source data.
- Article-to-shipment order linkage is feasible after grouped shipment parsing.

Known unknowns:

- Which date basis should be the default dashboard view when purchase and payment periods differ.
- Whether reports are intended for informal personal analysis or tax-grade output.
- Whether future Cardmarket exports will preserve the exact schema and language.
- Whether original files must be retained after import for auditability.
- Whether PII should be stored in clear text, masked, or optionally excluded.

Likely unknown unknowns / hidden risks:

- Old XLS encoding/codepage handling may corrupt German characters if the parser is chosen poorly; xlrd showed replacement characters where Excel showed correct umlauts.
- Future monthly exports may be incomplete sets: current source already has periods missing one or more expected XLS export perspectives.
- Duplicate detection can fail if dates/decimals are compared as raw strings instead of normalized business values.
- Product names are not immutable display metadata even when Product ID is stable.
- Shipment exports are not pure header tables; they combine shipment context and product-level continuation rows.

Tacit assumptions not proven by PRD:

- All months will have all four XLS perspectives for each direction.
- Shipment rows can be imported independently without forward-fill grouping.
- A product has one localized display name.
- CSV and XLS duplicates can be recognized by filename alone.
- Tax-oriented reports can be correct with the currently specified fields.

## Critical Findings

### Blockers

No hard source-data blocker was found. All 447 current source files were read successfully with tolerant XLS handling, and the core dashboard is buildable. There are, however, high-risk PRD gaps that should be fixed before implementation starts.

### High Risk

#### H1. Shipment exports are grouped rows; PRD does not specify forward-fill semantics

Evidence: Direct XLS profile shows 10,596 shipment raw rows, of which 8,432 have blank `OrderID` and `Username`. Example source: `PURCHASED SHIPMENTS-BYPAYMENTDATE-2016-06-01_2016-06-30.XLS`, rows 4-7 show one header row followed by continuation rows with blank order/party/date/cost fields but filled `Description`, `Product ID`, and `Localized Product Name`. The private OrderID is intentionally redacted.

Why it matters: A naive importer will either discard continuation rows, create orphan raw rows, or fail validation. This affects matching, raw auditability, product-level shipment descriptions, and plausibility checks.

Recommended action: Add an explicit PRD requirement: Shipment import must parse grouped records by carrying forward the previous nonblank `OrderID` and shipment header fields to continuation rows. Add `parent_order_id` or `resolved_order_id` to `shipment_export_row`, and make `source_row_number` mandatory.

Files/sources involved: PRD 8.1, 8.2, 9.1, 9.4; `Datenmodell.md` sections `shipment`, `shipment_export_row`; all `*SHIPMENTS*.XLS` files.

#### H2. MVP scope is too broad for implementation readiness

Evidence: PRD MVP includes import, import history, normalized explorer, global filters, dashboard KPI cards, time series, top lists, reports, detail pages, product pages, interactive charts, and report export across sections 5 and 9.2-9.7.

Why it matters: The highest implementation risk is data correctness, not charting. Building dashboard, reports, product pages, and interactive diagram drilldowns before the importer is proven increases rework.

Recommended action: Redefine MVP as Import + Normalization + Validation + Explorer + Shipment Detail + minimal KPI summary. Move interactive chart click-to-filter, saved reports/filters, PDF export, and advanced report types to later phases.

Files/sources involved: PRD sections 5, 9.2-9.7, 17, 19.

#### H3. Parser strategy is underspecified for old/corrupt-looking XLS and Unicode fidelity

Evidence: `xlrd.open_workbook()` without corruption tolerance fails on `PURCHASED ARTICLES-BYPAYMENTDATE-2016-06-01_2016-06-30.XLS` with `CompDocError: Workbook corruption: seen[2] == 4`; with `ignore_workbook_corruption=True` it reads the sheet. In a sampled private address, xlrd returned replacement characters while Excel COM preserved the original non-ASCII text.

Why it matters: PRD 13 says the parser must not require Excel Desktop, which is reasonable, but the implementation must still read this old XLS corpus correctly. A technically compatible library can still corrupt names/addresses.

Recommended action: Add parser acceptance fixtures: at least one corruption-tolerant file, one grouped shipment file, one German-address file, one CSV/XLS duplicate pair, and one high-row-count file. Require Unicode fidelity tests or choose a parser/conversion pipeline proven against these fixtures.

Files/sources involved: PRD 9.1, 13, 16; `PURCHASED ARTICLES-BYPAYMENTDATE-2016-06-01_2016-06-30.XLS`; `PURCHASED SHIPMENTS-BYPAYMENTDATE-2016-06-01_2016-06-30.XLS`.

#### H4. Date-basis completeness is asymmetric across periods

Evidence: Source set has missing paired perspectives, for example `PURCHASED 2024-06` and `PURCHASED 2026-02` missing `ARTICLES BYPURCHASEDATE`, `SOLD 2019-01` missing both purchase-date XLS views, and `SOLD 2025-08` missing payment-date XLS views. PURCHASED article purchase-date order IDs are 698 while payment-date order IDs are 740.

Why it matters: Reports by purchase date and payment date will not always be equally complete. If the dashboard silently switches basis or combines both, users may see inconsistent totals.

Recommended action: Add a data coverage model per period/direction/export perspective, show coverage warnings in reports, and define a default report basis. Make incomplete basis coverage a visible report caveat.

Files/sources involved: PRD 8.3, 9.6, 19; `Datenmodell.md` import rules; asymmetric files listed in the inventory above.

#### H5. Shipment `Description` and `Product ID` in shipment exports are line-like, not only shipment header fields

Evidence: Continuation rows in shipment exports have blank shipment header fields but nonblank `Description`, `Product ID`, and `Localized Product Name`. The source profile shows 8,432 continuation shipment rows.

Why it matters: The current normalized `shipment.description` field can only store one description if treated as header-only. That loses raw source detail and makes shipment export rows harder to reconcile with article rows.

Recommended action: Keep full `shipment_export_row` as raw staging with resolved order context. Treat `shipment.description` as optional summary only, or create `shipment_export_line`/`raw_shipment_item` for these rows. Do not rely on shipment `Product ID` as a single header-level product.

Files/sources involved: `Datenmodell.md` `shipment` and `shipment_export_row`; PRD 7, 9.4, 12; all `*SHIPMENTS*.XLS` files.

### Medium Risk

#### M1. Product localized name is not a stable one-to-one product attribute

Evidence: Direct article rows show 4,065 Product IDs; 63 Product IDs have multiple localized names, such as Product ID `273699` with `Charizard` and `Glurak`, and Product ID `273710` with `Venusaur` and `Bisaflor`.

Why it matters: A product detail page or product dimension that overwrites `localized_product_name` on upsert will produce unstable UI labels and misleading filters.

Recommended action: Store observed product labels separately or keep canonical name plus alternate localized names. Product filters should search both article and localized names.

Files/sources involved: PRD 8.4, 9.3, 9.5; `Datenmodell.md` `product`; article XLS files.

#### M2. Duplicate detection must normalize values, not compare raw strings

Evidence: The five 2026 SOLD ARTICLE CSV files are exact duplicates of same-month XLS after normalizing IDs and numeric fields. Raw comparison by date/number formatting would miss the overlap.

Why it matters: Idempotent import and CSV/XLS overlap handling will fail if the duplicate key uses raw cell text.

Recommended action: Define duplicate keys over canonical values: normalized IDs, parsed timestamps, Decimal amounts, normalized comments, source direction/level/basis. Treat CSV as alternate source when matching XLS exists.

Files/sources involved: PRD 8.5, 9.1, 16; `Datenmodell.md` import rule 8; `SOLD ARTICLES-BYPURCHASEDATE-2026-01..05` CSV and XLS files.

#### M3. Tax/controlling reports are underspecified

Evidence: PRD target usage includes `Reports fuer Steuer, Controlling`, but report requirements only list descriptive totals. Source has VAT Number, professional flag, country, costs, commission, and trustee fee, but no tax rules or account-owner tax treatment.

Why it matters: Users may trust reports beyond what the data model can legally/accountingly prove.

Recommended action: Reword MVP reports as operational/export summaries, not tax-compliant reports. Add tax-grade reporting only after explicit accounting rules are supplied.

Files/sources involved: PRD 2, 9.6, 14; shipment XLS files.

#### M4. Acceptance criteria are mostly outcome-based, not fixture-based

Evidence: PRD 19 says imports work, new months work, duplicates are avoided, and reports export. It does not name concrete fixture files or expected row/order counts.

Why it matters: Implementation can appear correct while failing continuation rows, encoding, CSV overlap, or asymmetric periods.

Recommended action: Add fixture-driven acceptance criteria using specific current files and expected counts from this review.

Files/sources involved: PRD 16, 19; all source XLS/CSV.

#### M5. Party/address normalization may be unnecessary for MVP and risky for privacy

Evidence: PRD includes `party` and `address`; source contains names, usernames, streets, cities, countries, and VAT numbers. Article files do not contain these fields; shipment continuation rows have blanks.

Why it matters: Normalizing addresses increases PII surface without clearly supporting the main dashboard job. It also requires grouped-row header context.

Recommended action: For MVP, store party/address as shipment snapshot fields and mask by default in UI. Defer separate `party` and `address` dimensions unless repeated-counterparty analysis is required.

Files/sources involved: PRD 9.3, 12, 14; `Datenmodell.md` `party`, `address`; shipment XLS files.

### Low Risk / Polish

#### L1. Interactive chart click-to-filter is a nice-to-have, not a readiness requirement

Evidence: PRD 9.7 requires clicking chart segments to set filters.

Why it matters: Adds UI state complexity before data correctness is proven.

Recommended action: Move click-to-filter to Phase 3 or 4. Keep hover tooltips and static filtered charts for MVP.

Files/sources involved: PRD 9.7, 11, 17.

#### L2. Saved reports and saved filters should remain outside MVP

Evidence: PRD 12 recommends `saved_report` and `saved_filter`, while PRD 17 places them in Phase 4.

Why it matters: These tables are speculative until report/filter behavior stabilizes.

Recommended action: Keep them out of MVP schema. Add later migration when needed.

Files/sources involved: PRD 12, 17.

#### L3. Performance target is plausible but not source-derived

Evidence: PRD 15 targets 100,000-250,000 article rows. Current source has 21,172 total rows and 10,576 article rows.

Why it matters: The target is fine, but tests should scale with generated fixtures, not current data only.

Recommended action: Add synthetic load tests after import correctness tests.

Files/sources involved: PRD 15; source profile summary.

## Efficiency / Architecture Notes

Simplest viable architecture:

1. Upload/import file.
2. Parse filename into import metadata.
3. Parse sheet/CSV rows into raw staging exactly as source rows, including `source_row_number`.
4. For Shipment exports, build `resolved_order_id` by forward-filling nonblank `OrderID` and header fields across continuation rows.
5. Normalize IDs, dates, decimals, booleans, and currency.
6. Upsert shipment headers by `direction + order_id`.
7. Upsert shipment events by `shipment_id + event_type`.
8. Upsert article lines from Article exports; link by `direction + shipment_nr`.
9. Use Shipment raw rows for validation/cross-checking, not as primary article-line source unless Article rows are absent.
10. Build read models/materialized views for dashboard totals.

Recommended MVP storage: SQLite is enough for the current size and a local private app, provided indexes exist on order IDs, product IDs, dates, direction, category, expansion, and username. PostgreSQL can wait unless remote/private-server hosting is chosen.

Avoidable complexity:

- Do not build a generalized BI/report builder first.
- Do not normalize addresses aggressively before privacy choices are settled.
- Do not create saved report/filter schemas until usage patterns are real.
- Do not let chart interaction drive the data model.
- Do not merge CSV and XLS using raw strings; normalize first.

## Open Questions for Human

1. Should dashboard defaults use purchase date, payment date, or let each widget choose? This changes event filters, report totals, and coverage warnings.
2. Are reports intended only for personal/operational analysis, or must any report be tax-grade? Tax-grade reports need accounting rules not present in the PRD.
3. Should CSV files be treated as fallback/verification when a matching XLS exists, or should the user choose preferred source? Current CSV rows duplicate XLS after normalization.
4. Should names, streets, cities, and VAT numbers be imported by default, masked by default, or excluded unless explicitly enabled?
5. Should original source files be stored/archived by the app after import, or is raw staging plus file metadata sufficient?
6. Are future Cardmarket exports expected to have the same English column names, or must the importer support localized/changed headers?
7. When purchase-date and payment-date export sets are incomplete for a period, should reports hide that period, warn only, or attempt fallback from the other basis?

## Final Recommendation

Proceed after specific PRD edits. Do not implement the PRD as-is.

Required edits before implementation:

1. Rewrite the import section to include grouped Shipment row parsing and forward-fill semantics.
2. Add parser acceptance fixtures for old XLS tolerance, Unicode fidelity, grouped shipment rows, CSV/XLS duplicate overlap, and asymmetric period coverage.
3. Reduce MVP scope to import, normalization, validation, explorer, shipment detail, product aggregate basics, and minimal KPIs.
4. Add explicit date-basis and coverage-warning behavior for reports.
5. Revise product modeling to allow multiple observed localized names per Product ID.
6. Reframe tax/controlling output as operational summaries unless accounting rules are supplied.
7. Make raw `source_row_number` mandatory for all staging rows and normalized line provenance.

After those edits, the data and model are sufficient for a practical MVP.
